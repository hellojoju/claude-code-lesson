Checkpoints（检查点）与回滚

你的重构刚刚破坏了 3 个功能。测试失败了。API 返回 500 错误。而且你无法准确记得改了什么。这就是 checkpoints 如何拯救你 —— 一个按键操作，你就回到了正常工作状态，准备好尝试更谨慎的方法。

Checkpoints 允许你保存会话状态并回滚到 Claude Code 会话中的先前时刻。这对于探索不同方法、从错误中恢复或比较替代方案非常有价值。

"糟糕"时刻

我们都经历过：

你: 重构认证模块以使用 OAuth2

Claude: [跨 12 个文件进行了 47 处更改]

你: 运行测试

Claude: 测试失败。23 个断言被破坏。
       支付处理出错。
       数据库连接丢失。

你: ...刚才发生了什么？

没有 checkpoints，你就陷入困境了。你可能要花几个小时调试。或者你可能不得不撤销一切重新开始，丢失所有混在错误中的正确更改。

有了 checkpoints： 
你: [按两次 Esc] 回滚到重构前

Claude: 已恢复到 checkpoint "auth-oauth2-start"。
       所有文件已还原。会话已恢复。

你: 让我们再试一次，一次一个模块。

这就是 checkpoints 的威力 —— 从任何"糟糕"时刻瞬间恢复。

什么是 Checkpoints？

Checkpoints 是会话状态的快照，包括：

组成部分 捕获内容
消息 交换的所有用户和助手消息
文件 会话期间进行的所有文件修改
工具 完整的工具调用历史
上下文 会话元数据和状态

把 checkpoints 想象成游戏中的"存档点"。在任何时刻，你都可以加载之前的存档并尝试不同的路径。

核心概念

概念 描述
Checkpoint 包括消息、文件和上下文的会话状态快照
Rewind（回滚） 返回到之前的 checkpoint，丢弃后续更改
Branch Point（分支点） 从该 checkpoint 探索多种方法的节点
Recovery Point（恢复点） 在危险操作前创建的 checkpoint

访问 Checkpoints

你可以通过两种主要方式访问和管理 checkpoints：

使用键盘快捷键

按两次 Esc（Esc + Esc）打开 checkpoint 界面并浏览已保存的 checkpoints。

使用 Slash 命令

使用 /rewind 命令（别名：/checkpoint）快速访问：

# 打开回滚界面
/rewind

# 或使用别名
/checkpoint

Checkpoint 策略决策树

何时应该创建 checkpoints？何时应该回滚？这个决策树指导你通过 checkpoint 工作流：

flowchart TD A[我想做出更改] --> B{是否有风险？} B -->|是| C[记录当前 checkpoint ID] B -->|否| D[正常进行] C --> E[进行更改] D --> E E --> F{成功了吗？} F -->|是| G[继续工作] F -->|否| H{想尝试不同方法？} H -->|是| I[按 Esc+Esc] H -->|否| J[调试问题] I --> K[选择较早的 checkpoint] K --> L{恢复什么？} L -->|全部| M[恢复代码和会话] L -->|仅文件| N[仅恢复代码] L -->|仅上下文| O[仅恢复会话] L -->|保留但压缩| P[从此处开始摘要] M --> E N --> E O --> E P --> Q[摘要替换消息] G --> R{想比较方法？} R -->|是| S[记录当前状态] S --> I R -->|否| T[继续会话]

回滚选项详解

当你回滚时，会看到五个选项。理解每个选项有助于你选择正确的恢复策略：

1. 恢复代码和会话

使用场景： 你想要完全重置到一个已知良好的状态。 
之前: Checkpoint A（正常状态）
      ↓
      已做出更改（损坏状态）
      
之后: Checkpoint A 已恢复
      文件：已还原
      消息：已还原
      工具历史：已还原

这是你的"核选项" —— 完全回滚到保存点。

2. 仅恢复会话

使用场景： 你想要重置上下文但保留文件更改。 
场景：你做出了正确的代码更改但会话
      变得混乱或走向了错误的方向。

结果：文件保持原样
      会话重置到 checkpoint
      你可以在干净的上下文中重新解释需求

3. 仅恢复代码

使用场景： 会话有帮助但代码更改破坏了某些东西。 
场景：Claude 搞清楚了架构但实现
      有 bug。

结果：文件还原到 checkpoint 状态
      会话历史保留
      你可以回顾推理过程

4. 从此处开始摘要

使用场景： 会话太长，你想压缩历史。 
场景：你已经工作了 2 小时，上下文
      变得臃肿，但不想丢失一切。

结果：Claude 生成会话摘要
      摘要替换从 checkpoint 开始的消息
      原始消息保留在转录中
      上下文窗口释放

你可以可选地提供指令来聚焦摘要：

用户: 从此处开始摘要，聚焦于 API 设计决策

5. 没关系

使用场景： 你不小心打开了回滚菜单或改变了主意。 
结果：返回当前状态，不做任何更改

自动 Checkpoints

Claude Code 自动为你创建 checkpoints：

触发条件 行为
每个用户提示 自动创建新 checkpoint
会话持久化 checkpoints 跨会话保存
自动清理 超过 30 天的 checkpoints 被删除

这意味着你始终可以回滚到会话中的任何先前时刻，从几分钟前到几天前。

配置

你可以在设置中切换自动 checkpoints：

{
  "autoCheckpoint": true
}
  • autoCheckpoint：启用或禁用每个用户提示上的自动 checkpoint 创建（默认：true）

上下文窗口与 Checkpoint 策略

理解上下文窗口如何工作，才能制定有效的 checkpoint 策略。

上下文窗口基础

每个 Claude 会话在一个上下文窗口内运行——模型可以同时持有的信息总量。

模型 上下文限制 约等于
Claude Opus 4.6 200,000 token ~150,000 词，200-300 页文本
Claude Sonnet 4.6 200,000 token 同上

上下文饱和症状：漂移

当上下文接近饱和时，会出现漂移现象：

早期会话：Claude 严格遵循你建立的约束
          "所有 API 响应必须包含 requestId"

漂移开始：Claude 开始做出不一致的决策
          [生成的 API 响应缺少 requestId]
          [使用了你之前明确禁止的库]
          [偏离了已批准的架构]

警告信号：
├── 重复提问相同信息
├── "忘记"之前的约束
├── 提出与已做决策矛盾的方案
└── 输出质量明显下降

50% 实践：黄金法则

经验丰富的 Claude Code 用户遵循一个惯例：当上下文使用达到 40-50% 时，准备切换。

flowchart LR A[0-40%: Plan 区域] --> B[40-60%: Code 区域] B --> C[60-80%: 警告区域] C --> D[80-100%: Dump 区域] A -->|安全规划复杂任务| E[✓ 正常工作] B -->|避免开始新长任务| F[⚠ 完成当前思路] C -->|准备切换| G[🔴 记录 checkpoint] D -->|立即停止| H[⛔ 回滚或新会话]

Checkpoint 策略详解

基于上下文窗口特性，制定策略性 checkpoint 计划：

策略 1：里程碑命名法

为重要 checkpoint 使用描述性名称：

Checkpoint 策略：
├── 功能开始："feature-auth-start"
├── 模型完成："feature-auth-models-done"
├── API 完成："feature-auth-api-done"
├── 测试通过："feature-auth-tests-passing"
└── 部署准备："feature-auth-deploy-ready"

好处：
├── 快速定位回滚点
├── 清晰的进度追踪
└── 团队协作时的明确参考

策略 2：危险操作前必存

在以下操作前记录 checkpoint：

操作类型 风险级别 Checkpoint 名称建议
数据库迁移 高 "pre-db-migration"
大规模重构 高 "pre-refactor-[module]"
删除代码 高 "pre-delete-[area]"
添加依赖 中 "pre-add-[package]"
配置更改 中 "pre-config-[service]"

策略 3：增量测试验证

每完成一个步骤后验证并记录：

正确流程：
1. 实现步骤 A
2. 运行相关测试
3. 测试通过 → Checkpoint："step-a-done"
4. 测试失败 → 回滚，修复，重试
5. 实现步骤 B
6. ...

避免的错误流程：
1. 实现步骤 A、B、C、D
2. 运行测试 → 失败
3. 不知道哪个步骤出错
4. 困惑地回滚到起点

Rewind 最佳实践

✅ 应该做的

  • 定期检查快照列表
   每 30 分钟或完成重要功能后：
   → 打开 checkpoint 浏览器
   → 确认最新 checkpoint 时间戳
   → 记住关键 checkpoint ID
  • 为重要操作添加描述
   你: 添加用户认证功能（重要里程碑）
   [checkpoint 会记录此描述，便于后续定位]
  • 回滚前先确认影响范围
   步骤：
   → 打开 checkpoint 浏览器
   → 选择目标 checkpoint
   → 查看会修改哪些文件
   → 确认后再执行回滚
  • Rewind 与 Git 配合使用
   Rewind：会话内的快速回退
   Git：长期版本控制和协作
   组合：Rewind 试错 → 成功后 Git commit

❌ 避免做的

  • 依赖 Rewind 替代 Git
   问题：Checkpoint 仅保留 30 天
   风险：历史记录永久丢失
   解决：重要更改必须提交 Git
  • 回滚后忘记验证
   错误：回滚后直接继续工作
   正确：回滚 → 运行测试 → 验证状态 → 继续
  • 跨会话期待 Rewind
   Checkpoint 仅在当前会话有效
   新会话开始后，旧 checkpoint 不可用
  • 忽略 Bash 命令的文件系统更改
   Bash 的 rm、mv、cp 不被 checkpoint 跟踪
   回滚后必须手动验证这些更改

会话恢复技巧

当需要跨会话保持连续性时，使用以下技巧：

四个连续性文档

文档 作用 Claude 自动读取
CLAUDE.md 项目约定和架构上下文 是，每个会话开始
PRD.md 正在构建什么 需要明确请求
README.md 当前实现状态 需要明确请求
progress.md 会话级笔记和决策 需要明确请求

新会话恢复模板

新会话开始提示：

新会话。按顺序读取以下文件：
1. CLAUDE.md
2. PRD.md
3. README.md（特别是实现状态部分）
4. progress.md

确认你对项目状态的理解并告诉我我们停在哪里。

实现状态追踪示例

## 实现状态

- ID: feature-auth-tests-passing
- 时间: 2024-01-15 14:30
- 状态: 所有测试通过，准备开始邮件通知

上下文压缩策略

当会话接近上下文上限时，使用"从此处开始摘要"压缩历史。

何时压缩

触发条件：
├── 上下文使用 > 50%
├── 开始注意到漂移症状
├── 准备开始新的大任务
└── Claude 重复提问已知信息

压缩选项使用

选择"从此处开始摘要"时，Claude 会：

1. 分析从 checkpoint 开始的所有消息
2. 生成聚焦摘要
3. 摘要替换原始消息序列
4. 原始消息保留在转录中（可导出）
5. 上下文窗口释放

聚焦摘要指令

可以提供指令让摘要聚焦特定内容：

从此处开始摘要，聚焦于：
- API 设计决策
- 数据库架构选择
- 测试策略约定

忽略：
- 临时调试讨论
- 失败尝试的细节

cc-context-stats 使用指南

cc-context-stats 是一个实时上下文监控工具，在状态栏显示你的上下文使用区域，帮助你决定何时 checkpoint 并切换会话。

安装

# 通过 npm 安装
npm install -g cc-context-stats

# 或克隆仓库本地安装
git clone https://github.com/luongnv89/cc-context-stats
cd cc-context-stats && npm install

区域指示器解读

区域 上下文范围 状态栏颜色 建议行动
Plan 0-50% 🟢 绿色 安全进行规划和复杂任务
Code 50-80% 🟡 黄色 避免开始新的长任务，完成当前工作
Dump 80-100% 🟠 橙色 立即完成当前思路，准备 checkpoint

实战使用流程

1. 启动复杂任务 → 区域显示 Plan (绿色)
   ↓ 安全规划
   
2. 工作进行中 → 区域变为 Code (黄色)
   ↓ 注意：不要启动新大任务
   
3. 接近上限 → 区域变为 Dump (橙色)
   ↓ 立即行动：
   a. 完成当前代码块
   b. 运行测试验证
   c. 记录 checkpoint ID
   d. 创建 progress.md 记录状态
   e. 开始新会话

与 Checkpoint 配合

sequenceDiagram participant U as 用户 participant C as Claude Code participant S as cc-context-stats participant CP as Checkpoint U->>S: 启动监控 S->>U: 显示 Plan 区域 (绿色) U->>C: 开始复杂任务 S->>U: 区域变为 Code (黄色) U->>C: 继续工作 S->>U: 区域变为 Dump (橙色) U->>C: 完成当前思路 C->>U: 任务完成 U->>CP: 记录 checkpoint U->>U: 创建 progress.md U->>C: 开始新会话

最佳配置

// 推荐的监控阈值配置
{
  "contextThresholds": {
    "planZone": 50,    // 0-50% 绿色
    "codeZone": 80,    // 50-80% 黄色
    "dumpZone": 100    // 80-100% 橙色
  },
  "alertOnZoneChange": true,  // 区域变化时提醒
  "showPercentage": true      // 显示具体百分比
}

Checkpoint 反模式

识别并避免这些常见的错误用法：

反模式 1：Checkpoint 作为唯一备份

问题：
├── Checkpoint 仅保留 30 天
├── 会话结束后可能无法访问
└── 不支持多人协作

正确做法：
重要更改 → 立即 Git commit
Checkpoint → 仅用于试错和临时回滚

反模式 2：无限回滚链

错误场景：
你: 回滚到 checkpoint A
你: 尝试方法 X → 失败 → 回滚到 checkpoint A
你: 尝试方法 Y → 失败 → 回滚到 checkpoint A
你: 尝试方法 Z → 失败 → 回滚到 checkpoint A
... 无限循环 ...

诊断：
├── 方法 A 本身可能有问题
├── 约束条件可能不完整
├── 需要调整策略，而不是无限回滚

正确做法：
最多尝试 3 种方法
如果都失败 → 分析根本原因 → 调整约束 → 再尝试

反模式 3：忽视外部更改

问题：
Checkpoint 不捕获以下更改：
├── Bash 的 rm、mv、cp 命令
├── 外部编辑器的文件修改
├── Git 操作
├── 环境变量更改

风险：
回滚后，这些更改仍然存在
可能造成状态不一致

正确做法：
回滚后 → 全面验证 → 检查外部更改状态

反模式 4：上下文饱和后继续工作

错误场景：
上下文 > 80% (Dump 区域)
↓
继续添加新任务
↓
Claude 输出质量下降
↓
产生错误代码
↓
花更多时间调试
↓
上下文继续增长
↓
恶性循环

正确做法：
Dump 区域触发 → 立即停止 → 记录 checkpoint → 新会话

反模式 5：过度细分 checkpoint

问题：
每 5 分钟创建一个 checkpoint
↓
checkpoint 列表过长
↓
难以找到正确的回滚点
↓
浪费时间浏览

正确做法：
仅在有意义的变化点创建 checkpoint：
├── 功能模块完成
├── 测试通过
├── 准备危险操作前
├── 发现需要回滚时

策略 详情
创建 每个用户提示自动创建
保留 最长 30 天
清理 自动删除旧的 checkpoints
存储 Claude Code 数据目录中的本地磁盘

工作流模式

用于探索的分支策略

当探索多种方法时：

1. 从初始实现开始 → Checkpoint A
2. 尝试方法 1 → Checkpoint B
3. 回滚到 Checkpoint A
4. 尝试方法 2 → Checkpoint C
5. 比较 B 和 C 的结果
6. 选择最佳方法并继续

安全重构模式

当做出重大更改时：

1. 当前状态 → Checkpoint（自动）
2. 开始重构
3. 运行测试
4. 如果测试通过 → 继续工作
5. 如果测试失败 → 回滚并尝试不同方法

最佳实践

由于 checkpoints 是自动创建的，你可以专注于工作，不必担心手动保存状态。然而，请记住这些实践：

有效使用 Checkpoints

应该：
  • 在回滚前审查可用的 checkpoints
  • 当你想探索不同方向时使用回滚
  • 保留 checkpoints 以比较不同方法
  • 理解每个回滚选项的作用（恢复代码和会话、恢复会话、恢复代码或摘要）
  • 在重要里程碑记录 checkpoint IDs
  • 记录每个 checkpoint 处做出的决策
不应该：
  • 仅依赖 checkpoints 进行代码保存
  • 期望 checkpoints 跟踪外部文件系统更改
  • 使用 checkpoints 作为 git 提交的替代
  • 回滚后忘记验证
  • 回滚后跳过测试

工作流最佳实践

  • 危险操作前： 记录你的 checkpoint ID
  • 里程碑后： 验证测试通过，记录 checkpoint
  • 探索时： 在每个 checkpoint 记录结果
  • 卡住时： 回滚到最后已知良好状态

限制

Checkpoints 有以下限制：

限制 影响 缓解措施
Bash 命令更改不被跟踪 rm、mv、cp 等文件系统操作不被捕获 谨慎使用；回滚后验证
外部更改不被跟踪 Claude Code 外部做出的更改不被捕获 将重要更改提交到 git
不是版本控制的替代 无协作、历史或分支功能 用 git 保存永久更改
会话范围存储 不为长期归档设计 将成功状态提交到 git

Checkpoints 不捕获的内容

不被捕获：
├── 修改文件系统的 Bash 命令（rm, mv, cp）
├── 在你的编辑器中做出的更改
├── 在 Claude Code 外的终端中做出的更改
├── Git 操作（commit, push, pull）
└── 环境变量更改

被捕获：
├── 所有用户消息
├── 所有助手响应
├── 通过 Edit/Write 工具做出的文件更改
├── 工具使用历史
└── 会话上下文

故障排查

Checkpoints 缺失

问题： 预期的 checkpoint 未找到 解决方案：
  • 检查 checkpoints 是否被清除（手动清理或设置）
  • 验证设置中 autoCheckpoint 已启用
  • 检查可用磁盘空间
  • 寻找时间戳略有不同的 checkpoints
调试步骤： 
你: /rewind

[Checkpoint 列表出现]

你: 我看不到 2 小时前的 checkpoint。

Claude: Checkpoints 按时间戳列出。 
        让我帮你找到它...

[滚动列表或按日期搜索]

回滚失败

问题： 无法回滚到 checkpoint 解决方案：
  • 确保没有未提交更改与 git 冲突
  • 检查 checkpoint 是否损坏（罕见）
  • 尝试回滚到不同的 checkpoint
  • 重启 Claude Code 并重试
错误恢复： 
用户: [尝试回滚]
错误：回滚失败 - 检测到文件冲突

用户: 我该怎么办？

Claude: 文件已在此会话外被修改。
        选项：
        1. 提交或暂存你的外部更改
        2. 尝试"仅恢复代码"以保留会话
        3. 尝试更早的 checkpoint

Checkpoint 损坏（罕见）

问题： Checkpoint 数据损坏或不完整 解决方案：
  • 尝试附近的 checkpoint（之前或之后）
  • 检查磁盘错误
  • 重启 Claude Code 会话
  • 作为最后手段，开始新会话

与 Git 的集成

Checkpoints 补充（但不替代）git：

功能 Git Checkpoints
范围 文件系统 会话 + 文件
持久性 永久 会话基础（30 天）
粒度 提交 任意点
速度 较慢（commit, push, pull） 即时
共享 是（远程仓库） 有限（导出）
分支 是 通过回滚/重试
历史 完整提交历史 仅近期会话
协作 多用户 单用户

推荐集成模式

1. 用 checkpoints 进行快速实验
   ↓
2. 当实验成功时，提交到 git
   ↓
3. 用 git 提交保存永久代码更改
   ↓
4. 用 git 分支进行长期功能
   ↓
5. 在 git 操作前创建 checkpoint
   ↓
6. 如果 git 操作失败，回滚 checkpoint

快速开始指南

基本工作流

  • 正常工作 - Claude Code 自动创建 checkpoints
  • 想回去？ - 按两次 Esc 或使用 /rewind
  • 选择 checkpoint - 从列表中选择以回滚
  • 选择恢复什么 - 从恢复代码和会话、恢复会话、恢复代码、从此处开始摘要或取消中选择
  • 继续工作 - 你回到了那个时刻

总结

Checkpoints 是 Claude Code 中的自动功能，让你安全探索不同方法而不必担心丢失工作。每个用户提示自动创建新 checkpoint，所以你可以回滚到会话中的任何先前时刻。

主要好处

好处 描述
无畏实验 尝试多种方法无风险
快速恢复 一个按键回到工作状态
比较方案 替代方案的并排评估
与 Git 集成 Checkpoints 用于实验，git 用于持久化

Checkpoint 思维模式

没有 CHECKPOINTS：
  做更改 → 希望成功 → 如果坏了调试 → 惊慌

有了 CHECKPOINTS：
  记录 checkpoint → 做更改 → 测试 → 
  如果成功：继续 → 
  如果失败：回滚 → 尝试不同方法

记住

  • Checkpoints 是自动的 —— 无需手动保存
  • 按两次 Esc 访问任何 checkpoint
  • 为你的情况选择正确的恢复选项
  • 用 checkpoints 进行快速实验
  • 用 git 提交保存永久代码更改
  • 最好的 checkpoint 是你在需要之前创建的那个
Checkpoints：因为当你需要探索多种未来时，"撤销"是不够的。
