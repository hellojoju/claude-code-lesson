代码审查流程

可靠性: A 来源: Claude Code 官方文档 + 行业最佳实践

代码审查价值

代码审查是保证代码质量的重要环节。Claude Code 可以辅助完成代码审查，提高审查效率和覆盖面，让人工审查专注于更有价值的方面。

AI 辅助审查

审查 Skill 配置

# skills/code-review/skill.yaml
id: code-review
name: Code Review Assistant
version: 1.0.0
description: AI 辅助代码审查

triggers:
  - type: command
    value: /review

execution:
  mode: single

prompts:
  system: |
    你是一个专业的代码审查助手。审查代码时关注：
    
    ## 代码质量
    - 代码是否清晰易读
    - 命名是否有意义
    - 函数是否职责单一
    - 是否有重复代码
    
    ## 潜在问题
    - 边界条件处理
    - 空值检查
    - 错误处理
    - 资源泄漏
    
    ## 安全问题
    - 输入验证
    - SQL 注入风险
    - XSS 漏洞
    - 敏感信息暴露
    
    ## 性能问题
    - 不必要的循环
    - 内存泄漏风险
    - 数据库查询优化
    
    ## 最佳实践
    - 是否遵循项目规范
    - 是否有更好的实现方式
    - 测试覆盖是否充分
    
    输出格式：
    1. 总体评价
    2. 问题列表（按严重程度分类）
    3. 改进建议

使用示例

# 审查当前文件
/review src/auth/login.ts

# 审查最近的变更
/review --diff HEAD~1

# 审查特定 PR
/review --pr 123

# 关注特定方面
/review --focus security src/api/

审查报告生成

报告模板

# 代码审查报告

## 基本信息
- PR: #123
- 作者: @developer
- 审查时间: 2024-01-15
- 变更文件: 5 个

## 自动化检查结果
| 检查项 | 状态 | 详情 |
|--------|------|------|
| Lint | ✅ 通过 | 无警告 |
| 类型检查 | ✅ 通过 | 无错误 |
| 单元测试 | ✅ 通过 | 15/15 通过 |
| 覆盖率 | ⚠️ 警告 | 75% (低于 80%) |

## AI 审查发现

### 🔴 严重问题 (1)
1. **SQL 注入风险** (src/api/users.ts:45)
   ```typescript
   // 问题代码
   const query = `SELECT * FROM users WHERE id = ${userId}`;
   
   // 建议修改
   const query = 'SELECT * FROM users WHERE id = ?';
   db.query(query, [userId]);
   ```

### 🟡 一般问题 (2)
1. **缺少错误处理** (src/services/auth.ts:23)
   - 建议添加 try-catch 处理异常情况

2. **魔法数字** (src/utils/calculate.ts:12)
   - 建议将数字 `86400` 提取为常量

### 🔵 建议改进 (3)
1. 考虑使用可选链操作符简化代码
2. 函数 `processData` 可以拆分为更小的函数
3. 建议添加更多单元测试覆盖边界情况

## 人工审查要点
- [ ] 确认业务逻辑正确性
- [ ] 评估对现有功能的影响
- [ ] 检查是否有遗漏的测试用例

## 总结
代码整体质量良好，但存在一个安全问题需要修复后再合并。
建议作者处理上述问题后重新提交审查。

审查反馈技巧

建设性反馈示例

# 好的反馈 ✅

"这里使用 `any` 类型会失去类型检查的好处。
建议定义一个接口来描述数据结构：

```typescript
interface UserData {
  id: string;
  name: string;
  email: string;
}

function processUser(data: UserData) {
  // ...
```

这样可以获得更好的类型安全和 IDE 支持。"

# 不好的反馈 ❌

"不要用 any，这样不好。"

反馈分类

  • 必须修改：影响功能或有安全风险
  • 建议修改：提升代码质量
  • 讨论：需要进一步沟通的设计决策
  • 称赞：肯定好的实现

审查效率优化

⏰ 时间管理

  • 单次审查不超过 1 小时
  • 大 PR 拆分为多个小 PR
  • 及时响应审查请求

🎯 关注重点

  • 让 AI 处理机械性检查
  • 人工关注业务逻辑
  • 优先处理严重问题

📝 文档先行

  • PR 描述清晰完整
  • 复杂逻辑添加注释
  • 提供测试步骤

🔄 持续改进

  • 记录常见问题模式
  • 更新审查检查清单
  • 定期分享审查经验
