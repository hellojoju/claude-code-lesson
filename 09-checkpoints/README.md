# Checkpoints（检查点）与回滚

## 为什么需要这个？

"我在做重要的改动，能不能随时保存进度？"

这是每个 Claude Code 用户都会遇到的问题：

```
场景：重构认证模块

你: 重构认证模块以使用 OAuth2

Claude: [跨 12 个文件进行了 47 处更改]

你: 运行测试

Claude: 测试失败。23 个断言被破坏。
       支付处理出错。
       数据库连接丢失。

你: ...刚才发生了什么？怎么回到正常状态？
```

没有 Checkpoints，你可能要花几个小时调试，或者不得不撤销一切重新开始，丢失所有混在错误中的正确更改。

**有了 Checkpoints：**

```
你: [按两次 Esc] 回滚到重构前

Claude: 已恢复到 checkpoint "auth-oauth2-start"。
       所有文件已还原。会话已恢复。

你: 让我们再试一次，一次一个模块。
```

这就是 Checkpoints 的威力 —— 一键从任何"糟糕"时刻恢复。

---

## 核心概念

### Checkpoint 是什么？

Checkpoint 是会话状态的完整快照：

| 组成部分 | 捕获内容 |
|----------|----------|
| **消息** | 交换的所有用户和助手消息 |
| **文件** | 会话期间进行的所有文件修改 |
| **工具** | 完整的工具调用历史 |
| **上下文** | 会话元数据和状态 |

把 Checkpoints 想象成游戏中的"存档点"——在任何时刻，你都可以加载之前的存档并尝试不同的路径。

### 关键术语

| 概念 | 描述 |
|------|------|
| **Checkpoint** | 会话状态快照（自动创建） |
| **Rewind（回滚）** | 返回到之前的 checkpoint |
| **Branch Point** | 从该 checkpoint 探索多种方法的节点 |
| **Recovery Point** | 在危险操作前创建的安全点 |

### 自动创建机制

Claude Code 自动为你创建 Checkpoints：

| 触发条件 | 行为 |
|----------|------|
| 每个用户提示 | 自动创建新 checkpoint |
| 会话持久化 | checkpoints 跨会话保存 |
| 自动清理 | 超过 30 天的 checkpoints 被删除 |

这意味着你**始终可以回滚**到会话中的任何先前时刻。

## 场景 1：重要改动前创建快照

### 问题场景

你即将进行一个可能有风险的操作——重构数据库层、修改核心配置、或尝试一个新的架构方案。你想确保如果出了问题，能快速恢复。

### 解决方案

虽然 Checkpoints 是自动创建的，但养成**记录当前 checkpoint** 的习惯能让你更快定位：

```
你: 我要在 checkpoint "pre-db-refactor" 开始数据库重构
    当前状态：所有测试通过，准备修改连接池配置

Claude: 已记录你的起始点。
       Checkpoint ID: pre-db-refactor
       时间: 2026-04-06 14:30
       
       现在可以安全地进行更改。
```

### 操作步骤

1. **确认当前状态良好**
   ```
   你: 运行测试，确认一切正常
   
   Claude: 所有 127 个测试通过。
   ```

2. **记录 checkpoint（心理标记）**
   - 在你的消息中描述当前状态
   - Claude 会自动创建 checkpoint
   - 记住这个时刻的描述

3. **进行风险操作**
   ```
   你: 重构数据库连接池，增加最大连接数到 50
   
   Claude: [修改配置文件]
           已更新 database.yml
           已修改 connection_pool.rb
   ```

4. **验证结果**
   ```
   你: 运行测试
   
   Claude: 3 个测试失败 —— 连接超时错误
   ```

5. **如需恢复，快速定位**
   - 按 `Esc` + `Esc` 打开 checkpoint 浏览器
   - 找到"pre-db-refactor"时刻的 checkpoint
   - 选择恢复选项

### 最佳实践：危险操作清单

在以下操作前特别留意：

| 操作类型 | 风险级别 | 建议描述 |
|----------|----------|----------|
| 数据库迁移 | 高 | "pre-db-migration" |
| 大规模重构 | 高 | "pre-refactor-[模块名]" |
| 删除代码 | 高 | "pre-delete-[区域]" |
| 添加依赖 | 中 | "pre-add-[包名]" |
| 配置更改 | 中 | "pre-config-[服务]" |

## 场景 2：回退到之前的状态

### 问题场景

你刚刚做了一些更改，结果出问题了——测试失败、功能破坏、或者代码走向了错误的方向。你需要快速回到之前的状态。

### 解决方案

使用 Rewind 功能回滚到之前的 checkpoint。

### 访问方式

**键盘快捷键：** 按两次 `Esc`（`Esc` + `Esc`）

**Slash 命令：** `/rewind` 或 `/checkpoint`

```
你: /rewind

Claude: [显示 checkpoint 列表]
       
       ┌─────────────────────────────────────┐
       │ 可用 Checkpoints                     │
       ├─────────────────────────────────────┤
       │ 14:35 - "数据库重构完成"             │
       │ 14:30 - "pre-db-refactor" ← 起点     │
       │ 14:15 - "添加用户认证"               │
       │ 14:00 - "会话开始"                   │
       └─────────────────────────────────────┘
       
       选择要回滚的 checkpoint...
```

### 五种恢复选项

当你选择一个 checkpoint 后，会看到五种恢复选项：

#### 选项 1：恢复代码和会话

**使用场景：** 完全重置到一个已知良好的状态。

```
效果：
├── 文件：已还原到 checkpoint 状态
├── 消息：已还原
├── 工具历史：已还原
└── 结果：干净的起点，可以重新开始
```

这是你的"核选项"——完全回滚到保存点。

#### 选项 2：仅恢复会话

**使用场景：** 文件更改是对的，但会话变得混乱或走向了错误方向。

```
效果：
├── 文件：保持当前状态
├── 会话：重置到 checkpoint
└── 结果：干净的上下文，可以重新解释需求
```

#### 选项 3：仅恢复代码

**使用场景：** 会话有帮助（比如推理过程），但代码更改破坏了某些东西。

```
效果：
├── 文件：还原到 checkpoint 状态
├── 会话：保持当前历史
└── 结果：可以回顾之前的推理过程
```

#### 选项 4：从此处开始摘要

**使用场景：** 会话太长，上下文臃肿，但不想丢失一切。

```
效果：
├── Claude 生成会话摘要
├── 摘要替换从 checkpoint 开始的消息
├── 原始消息保留在转录中
└── 上下文窗口释放
```

可以提供聚焦指令：

```
你: 从此处开始摘要，聚焦于 API 设计决策
```

#### 选项 5：没关系

**使用场景：** 不小心打开了菜单或改变了主意。

```
效果：返回当前状态，不做任何更改
```

### 实战演示

```
场景：重构导致测试失败

你: [按 Esc+Esc]

Claude: 显示 checkpoint 浏览器

你: 选择 "pre-db-refactor" checkpoint

Claude: 此 checkpoint 详情：
       - 时间：14:30
       - 修改文件：0（安全点）
       - 测试状态：全部通过

你: 选择 "恢复代码和会话"

Claude: 已恢复！
       - 所有文件已还原
       - 会话重置到重构前
       - 准备重新开始

你: 运行测试确认

Claude: 所有 127 个测试通过。状态正常。
```

## 场景 3：比较不同版本

### 问题场景

你有两种可能的实现方案，想比较它们的效果——比如比较 Redis 缓存 vs 内存缓存，或者比较两种不同的 API 设计。

### 解决方案

使用 Checkpoints 的"分支探索"模式，在同一起点尝试多种方案。

### 工作流程

```mermaid
flowchart LR
    A[记录起点] --> B[实现方案 A]
    B --> C[记录结果 A]
    C --> D[回滚到起点]
    D --> E[实现方案 B]
    E --> F[记录结果 B]
    F --> G[比较选择]
```

### 实战演示

**步骤 1：记录起点**

```
你: 我要比较两种缓存策略。
    当前 checkpoint 作为基线："cache-comparison-base"

Claude: 已记录基线。
       当前状态已保存。
```

**步骤 2：实现方案 A**

```
你: 用 Redis 实现缓存

Claude: [实现 Redis 缓存]
       - 添加 redis gem
       - 创建 cache_service.rb
       - 配置连接参数

你: 运行性能基准测试

Claude: 平均响应时间：45ms
       内存占用：+15MB
```

**步骤 3：记录方案 A 的结果**

```
你: 方案 A（Redis）结果：
    - 响应时间：45ms
    - 内存：+15MB
    - 复杂度：低

Claude: 已记录。Checkpoint: "redis-cache-complete"
```

**步骤 4：回滚到起点**

```
你: [按 Esc+Esc]
    选择 "cache-comparison-base"
    选择 "恢复代码和会话"

Claude: 已恢复到基线。
       Redis 实现已清除。
```

**步骤 5：实现方案 B**

```
你: 用内存缓存实现

Claude: [实现内存缓存]
       - 创建 in_memory_cache.rb
       - 使用 LRU 算法

你: 运行相同的性能基准测试

Claude: 平均响应时间：12ms
       内存占用：+8MB
```

**步骤 6：比较并选择**

```
你: 方案 B（内存缓存）结果：
    - 响应时间：12ms（快 3 倍）
    - 内存：+8MB（更低）
    - 复杂度：中

决策：选择方案 B —— 性能更好，资源占用更低

Claude: 确认选择方案 B。
       Checkpoint: "memory-cache-final"
```

### 适用场景

| 场景 | 比较内容 |
|------|----------|
| 性能优化 | 不同算法/策略的性能指标 |
| 架构决策 | 不同架构方案的优劣 |
| UI 设计 | 不同组件样式的视觉效果 |
| 技术选型 | 不同库/框架的功能对比 |

## 常见问题

### Q1: Checkpoint 能保存多久？

**A:** Checkpoints 默认保留 30 天，之后自动清理。重要更改应提交到 Git 进行永久保存。

### Q2: Bash 命令的更改会被跟踪吗？

**A:** 不会。`rm`、`mv`、`cp` 等文件系统操作不被 checkpoint 跟踪。回滚后需手动验证这些更改。

```
不被捕获：
├── Bash 的 rm、mv、cp 命令
├── 外部编辑器的修改
├── Git 操作
└── 环境变量更改
```

### Q3: Checkpoint 能替代 Git 吗？

**A:** 不能。它们是互补的：

| 功能 | Git | Checkpoints |
|------|-----|-------------|
| 持久性 | 永久 | 30 天 |
| 速度 | 较慢 | 即时 |
| 协作 | 支持 | 单用户 |
| 粒度 | 提交级别 | 任意时刻 |

**推荐模式：** Checkpoint 试错 → 成功后 Git commit

### Q4: 为什么找不到之前的 checkpoint？

**A:** 可能原因：
1. Checkpoints 超过 30 天被清理
2. 开启了新会话（checkpoint 仅在当前会话有效）
3. `autoCheckpoint` 设置被关闭

检查设置：
```json
{
  "autoCheckpoint": true  // 确保已启用
}
```

### Q5: 回滚后测试还是失败怎么办？

**A:** 可能选择了错误的 checkpoint 或根本原因未解决：

1. 选择更早的 checkpoint 尝试
2. 回滚后运行测试验证状态
3. 分析根本原因再继续

### Q6: "从此处开始摘要" 会丢失信息吗？

**A:** 不会。原始消息保留在转录中，只是被压缩的摘要替换在主会话中。你可以导出完整转录查看细节。

## 上下文窗口与 Checkpoint 策略

### 为什么这很重要？

理解上下文窗口能帮助你制定有效的 checkpoint 策略。

### 上下文窗口基础

每个 Claude 会话在**上下文窗口**内运行——模型可以同时持有的信息总量。

| 区域 | 范围 | 建议 |
|------|------|------|
| **Plan** | 0-50% | 安全进行复杂规划 |
| **Code** | 50-80% | 避免开始新的长任务 |
| **Dump** | 80-100% | 立即完成，准备切换 |

### 漂移症状

当上下文接近饱和时会出现**漂移**：

```
警告信号：
├── Claude 重复提问相同信息
├── "忘记"之前的约束
├── 提出与已做决策矛盾的方案
└── 输出质量明显下降
```

### 黄金法则：50% 时准备切换

经验丰富的用户遵循：当上下文达到 **40-50%** 时，准备 checkpoint 并切换新会话。

### 使用 cc-context-stats 工具

实时监控上下文使用：

```bash
npm install -g cc-context-stats
```

状态栏显示当前区域（绿色 Plan / 黄色 Code / 橙色 Dump），帮助你决定何时 checkpoint。

## Checkpoint 反模式

识别并避免这些常见错误：

### 反模式 1：Checkpoint 作为唯一备份

```
错误：仅依赖 checkpoint 保存代码
风险：30 天后永久丢失
正确：重要更改必须 Git commit
```

### 反模式 2：无限回滚循环

```
错误：尝试方法 A → 失败 → 回滚 → 方法 B → 失败 → 回滚 → 方法 C → ...
诊断：起点本身可能有问题，或约束不完整
正确：最多 3 种方法，都失败则分析根本原因
```

### 反模式 3：忽视外部更改

```
错误：回滚后直接继续工作
风险：Bash 命令的更改仍然存在，可能状态不一致
正确：回滚 → 全面验证 → 检查外部更改状态
```
